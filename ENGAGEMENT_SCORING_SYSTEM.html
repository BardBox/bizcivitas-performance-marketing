Section 01

System Overview

An automated lead intelligence system that tracks visitor behavior and assigns pipeline stages dynamically.

What It Does

This system tracks user engagement on the BizCivitas performance marketing website and dynamically assigns an engagement score based on user behavior. Based on the score, users are automatically categorized into a pipeline stage.

Key Principle: The "Converted" stage is determined by payment/membership data, not by engagement score. Payment always overrides score-based stages.

System Goals

  • Track user engagement across the website in real-time
  • Assign scores based on user actions (page visits, scroll depth, form interactions, etc.)
  • Automatically update lead pipeline stage based on accumulated score
  • Identify highly interested users before they convert
  • Detect converted users via Razorpay payment records
  • Allow admin to customize scoring values and pipeline thresholds
  • Support custom event categories added by admin
  • Maintain real-time engagement analytics in admin panel

Section 02

Pipeline Stages

Users automatically flow through 5 stages based on their engagement score.

NEW
COLD
WARM
HOT
CONVERTED
Stage Score Range Description Trigger
New 0 – 10 First interaction with the platform Automatic (default stage)
Cold 11 – 25 Low engagement visitor, browsing Score crosses cold threshold
Warm 26 – 50 Interested visitor, exploring content Score crosses warm threshold
Hot 51+ Highly engaged potential lead Score crosses hot threshold
Converted Payment detected User has purchased membership Razorpay payment verified

Important Rule: Converted users override ALL engagement scores. Once payment is verified, the pipeline stage becomes "Converted" regardless of the score value.

Pipeline Assignment Logic

function getPipelineStage(score, isConverted) { if (isConverted) return "converted"; const thresholds = await getThresholdsFromDB(); // { cold: 11, warm: 26, hot: 51 } if (score < thresholds.cold) return "new"; // 0-10 if (score < thresholds.warm) return "cold"; // 11-25 if (score < thresholds.hot) return "warm"; // 26-50 return "hot"; // 51+ }

Thresholds are configurable from the admin panel at /admin/scoring. The values (11, 26, 51) are defaults that can be adjusted per business needs.

Section 05

Frontend Tracking

How user behavior is captured on the website before and after inquiry creation.

Tracking Hook: useEngagementTracker

A React hook that automatically tracks all user behavior. It runs on every page and stores events in sessionStorage until the inquiry is created.

EventWhere It's TrackedHow

form_startedInquiryModalonFocus on form (first time only)
form_submittedInquiryModalAfter successful form submit

Storage Strategy

Before inquiry exists: All events are stored in sessionStorage as an array. No API calls are made to avoid creating orphan data.

After form submit: The syncEventsToInquiry(inquiryId) function sends ALL accumulated events to the backend in a single batch API call. Then sessionStorage is cleared.

Session ID

A unique session ID is generated per browser session and stored in sessionStorage. Format: ses_1710590400000_a7b3c9d

Section 06

Backend Processing

How the backend receives events, calculates scores, and assigns pipeline stages.

Score Calculation

The backend reads scoring values from the PmScoringConfig collection (not hardcoded). This allows admin to change scores without code deployment.

// Build score map from DB config const getScoreMap = async () => { const config = await PmScoringConfig.findOne(); const map = {}; for (const e of config.events) { if (e.isActive) map[e.key] = e.score; } return map; // { page_visit: 2, return_visit: 5, ... } };

Sync Process (POST /engagement/sync)

  1. Receive inquiryId + events[] array from frontend
  2. Fetch PmScoringConfig from DB to get current score values
  3. Loop through each event, look up score from config
  4. Add each event to inquiry.activityLog
  5. Accumulate inquiry.engagementScore
  6. Calculate pipeline stage using thresholds from DB
  7. Auto-update inquiry.status to match pipeline stage
  8. Save inquiry document

Status Auto-Update Rules

Current StatusNew Pipeline StageAction
newcold / warm / hotStatus updates to match
contactedanyNo auto-update (manual override)
convertedanyNever downgraded

Section 10

Score Decay & Maintenance

Preventing inactive users from staying in Hot stage forever.

How Score Decay Works

SettingDefaultDescription
EnabledYesScore decay is active
Inactive Days30Days without activity before decay triggers
Decay Amount10Points subtracted per decay cycle

Decay Logic

// Finds all non-converted inquiries inactive for 30+ days // Reduces their score by 10 (minimum 0) // Recalculates pipeline stage Example: Hot lead (score: 55) inactive 30 days → Score becomes 45 → Pipeline: Hot → Warm Warm lead (score: 30) inactive 30 days → Score becomes 20 → Pipeline: Warm → Cold

Converted leads are never decayed. Once a user pays, their status remains "Converted" permanently regardless of activity or score.

Running Decay

Score decay can be triggered via:

  • API call: POST /api/v1/pm/engagement/decay
  • Cron job: Set up a daily/weekly cron to call the decay endpoint
  • Manual: Can be triggered from admin panel in future
